2. SYSTEM ARCHITECTURE OVERVIEW
================================================================================

2.1 HIGH-LEVEL SYSTEM ARCHITECTURE

The system follows a 3-layer architecture:

[SENSORS]
   ↓ (Read environmental data)
[CONTROLLER (Greenhouse)]
   ↓ (Make decisions based on thresholds)
[ACTUATORS]
   ↓ (Execute control commands)
[PLANTS]
   ↓ (Respond to control)
[LOGGER]
   → (Persist data to CSV files)

2.2 COMPONENT RESPONSIBILITIES

SENSORS (Input Layer):
- Read environmental measurements
- Simulate realistic sensor variations
- Return values within specified ranges
- Timestamp all readings

CONTROLLER (Decision Layer):
- Read all sensor values
- Compare against thresholds
- Decide which actuators to activate
- Monitor plant health
- Generate alerts

ACTUATORS (Output Layer):
- Execute control commands
- Track power consumption
- Log activation events
- Support variable power levels (0-100%)

PLANTS (Monitor Layer):
- Track environmental conditions
- Update health status based on conditions
- Monitor growth progress
- Respond to actuator actions

LOGGER (Persistence Layer):
- Write sensor readings to CSV
- Record actuator events
- Track plant status changes
- Log system alerts
- Support data analysis

2.3 SYSTEM COMPOSITION (HAS-A Relationships)

Greenhouse (Main Container)
├── Vector of 5 Sensor pointers
├── Vector of 5 Actuator pointers
├── Vector of 3 Plant pointers
├── Queue of Alerts (FIFO)
└── Logger instance

This composition structure demonstrates the principle that a Greenhouse "HAS-A"
collection of sensors, actuators, and plants, rather than "IS-A" sensor or
actuator.

3.1 ABSTRACTION
================================================================================

Definition: Abstraction hides complex implementation details and exposes only
essential features through a simplified interface.

IMPLEMENTATION IN PROJECT:

1. SENSOR ABSTRACTION:
   ┌─────────────────────────────────────────┐
   │ class Sensor (Abstract Base Class)      │
   ├─────────────────────────────────────────┤
   │ protected:                              │
   │   string sensorID;                      │
   │   float currentValue;                   │
   │   float minRange, maxRange;             │
   ├─────────────────────────────────────────┤
   │ public:                                 │
   │   virtual float readValue() = 0;        │
   │   virtual string getSensorType() = 0;   │
   │   float getCurrentValue() const;        │
   └─────────────────────────────────────────┘

The abstract Sensor class defines the interface for all sensors without
specifying HOW each sensor reads its value. Each derived class implements the
readValue() method differently:

- MoistureSensor::readValue() → Simulates soil moisture reading
- TemperatureSensor::readValue() → Simulates temperature reading
- LightSensor::readValue() → Simulates light intensity reading
- HumiditySensor::readValue() → Simulates humidity reading
- WaterLevelSensor::readValue() → Simulates water tank level

CLIENT CODE DOESN'T NEED TO KNOW:
- How each sensor simulates readings
- The specific sensor type
- The internal calculation logic

CLIENT CODE ONLY USES:
- sensor->readValue()
- sensor->getSensorType()
- sensor->getCurrentValue()

2. ACTUATOR ABSTRACTION:
   ┌──────────────────────────────────────────┐
   │ class Actuator (Abstract Base Class)     │
   ├──────────────────────────────────────────┤
   │ protected:                               │
   │   string actuatorID;                     │
   │   bool isActive;                         │
   │   int powerLevel;                        │
   ├──────────────────────────────────────────┤
   │ public:                                  │
   │   virtual void activate(int level) = 0;  │
   │   virtual void deactivate() = 0;         │
   │   virtual string getActuatorType() = 0;  │
   └──────────────────────────────────────────┘

Each actuator type (WaterPump, Fan, Heater, Light, Sprinkler) implements its
own activation logic without client code knowing the details.

3. PLANT ABSTRACTION:
The Plant abstract class defines what all plants do:
- Update conditions based on environment
- Track health status
- Monitor water needs
- Check temperature tolerance

Each plant type (TomatoPlant, LettucePlant, MaizePlant) has different:
- Moisture requirements
- Temperature tolerances
- Light requirements
- Growth periods

The abstract interface hides these differences from the Greenhouse controller.

BENEFITS OF ABSTRACTION:
✓ Reduces complexity
✓ Hides implementation details
✓ Makes code more maintainable
✓ Allows future extensions without changing existing code
✓ Enforces contracts through pure virtual methods


3.2 POLYMORPHISM
================================================================================

Definition: Polymorphism allows objects of different types to be treated through
the same interface, with different behaviors at runtime.

IMPLEMENTATION IN PROJECT:

1. VIRTUAL METHOD DISPATCH - SENSORS:

Code:
   vector<Sensor*> sensors;  // Collection of base class pointers
   sensors.push_back(new MoistureSensor());
   sensors.push_back(new TemperatureSensor());
   sensors.push_back(new LightSensor());

   // Same method call → Different implementations
   for (auto sensor : sensors) {
       float value = sensor->readValue();  // Virtual method call
       logger.logSensorReading(sensor);
   }

At Runtime:
   sensors[0]->readValue() → Calls MoistureSensor::readValue()
   sensors[1]->readValue() → Calls TemperatureSensor::readValue()
   sensors[2]->readValue() → Calls LightSensor::readValue()

Each sensor returns a DIFFERENT value calculated using DIFFERENT logic, yet
the client code calls the SAME METHOD.

This is the power of polymorphism:
- ONE method call
- MULTIPLE different behaviors
- Determined at RUNTIME based on actual object type
- Client code doesn't care which sensor type it is

2. VIRTUAL METHOD DISPATCH - ACTUATORS:

Code:
   vector<Actuator*> actuators;
   actuators.push_back(new WaterPump());
   actuators.push_back(new CoolingFan());
   actuators.push_back(new Heater());

   // Same method, different behaviors
   actuators[0]->activate(80);   → Pump dispenses 0.40L water
   actuators[1]->activate(100);  → Fan starts cooling
   actuators[2]->activate(100);  → Heater starts heating

3. VIRTUAL METHOD DISPATCH - PLANTS:

Code:
   vector<Plant*> plants;
   plants.push_back(new TomatoPlant());
   plants.push_back(new LettucePlant());
   plants.push_back(new MaizePlant());

   // Same method, different implementations
   for (auto plant : plants) {
       plant->updateConditions(moisture, temp, light);
   }

Each plant evaluates conditions DIFFERENTLY:
- TomatoPlant: Requires 50-70% moisture
- LettucePlant: Requires 40-60% moisture
- MaizePlant: Requires 30-50% moisture (drought tolerant)

4. RUNTIME POLYMORPHISM MECHANISM:

The C++ compiler creates a Virtual Method Table (VTABLE) for each class:

   Sensor VTABLE:
   ├── readValue() → pointer to base implementation
   ├── getSensorType() → pointer to base implementation
   └── ...

   MoistureSensor VTABLE:
   ├── readValue() → pointer to MoistureSensor::readValue()
   ├── getSensorType() → pointer to MoistureSensor::getSensorType()
   └── ...

When sensor->readValue() is called, the program:
1. Looks up the actual runtime type
2. Finds the correct VTABLE entry
3. Calls the appropriate implementation

HOW POLYMORPHISM IMPROVES THIS PROJECT:
✓ Can add new sensor types without changing existing code
✓ Same control logic works for all sensor types
✓ Easy to add new actuators (Fan, Heater, etc.)
✓ Plants of different types coexist in one collection
✓ The Greenhouse class doesn't need to know specific types
✓ Demonstrates the Open/Closed Principle (open for extension, closed for modification)


3.3 INHERITANCE
================================================================================

Definition: Inheritance allows classes to inherit properties and methods from
parent classes, creating a hierarchy of related classes.

IMPLEMENTATION IN PROJECT:

1. SENSOR INHERITANCE HIERARCHY:

                    ┌────────────────────┐
                    │ Sensor (Abstract)  │
                    │ (Base Class)       │
                    └─────────┬──────────┘
                              │
                 ┌────────────┼────────────┐
                 │            │           │
          ┌──────▼─────┐ ┌───▼──────┐ ┌──▼───────┐
          │ Moisture   │ │Temperat.│ │Humidity │ ...
          │ Sensor     │ │ Sensor  │ │ Sensor  │
          └────────────┘ └─────────┘ └─────────┘

Each derived class inherits from Sensor:
- Inherits protected member variables (sensorID, currentValue, minRange, etc.)
- Inherits public methods (getSensorID(), getCurrentValue())
- Overrides pure virtual methods (readValue())
- Adds specific implementation for its type

Example - MoistureSensor:
   class MoistureSensor : public Sensor {
   public:
       MoistureSensor(string id = "MOIST_001");
       float readValue() override;
       string getSensorType() const override { 
           return "MoistureSensor"; 
       }
   };

MoistureSensor inherits:
- string sensorID
- float currentValue
- Methods like getCurrentValue(), getLastReadTime()

MoistureSensor specializes:
- readValue() with moisture-specific logic
- getSensorType() returns "MoistureSensor"

2. ACTUATOR INHERITANCE HIERARCHY:

                    ┌─────────────────────┐
                    │ Actuator (Abstract) │
                    │ (Base Class)        │
                    └──────────┬──────────┘
                               │
               ┌───────────────┼──────────────┐
               │               │              │
          ┌────▼──────┐  ┌─────▼────┐  ┌────▼──────┐
          │Water      │  │ Cooling  │  │  Heater   │ ...
          │ Pump      │  │  Fan     │  │           │
          └───────────┘  └──────────┘  └───────────┘

Each actuator inherits activate() and deactivate() interface but implements
them differently:
- WaterPump: Dispenses water
- CoolingFan: Cools the environment
- Heater: Heats the environment
- GrowLight: Provides artificial lighting

3. PLANT INHERITANCE HIERARCHY:

                    ┌────────────────────┐
                    │ Plant (Abstract)   │
                    │ (Base Class)       │
                    └─────────┬──────────┘
                              │
                   ┌──────────┼──────────┐
                   │          │         │
              ┌────▼──┐  ┌────▼───┐ ┌──▼────┐
              │Tomato │  │Lettuce │ │Maize  │
              │Plant  │  │ Plant  │ │Plant  │
              └───────┘  └────────┘ └───────┘

Each plant type inherits:
- updateConditions() method signature
- needsWater(), needsCooling(), needsHeating() methods
- Health tracking (isHealthy, currentMoisture, etc.)

But each specializes with different thresholds:
- TomatoPlant: minMoisture=50%, optimalTemperature=25°C
- LettucePlant: minMoisture=40%, optimalTemperature=20°C
- MaizePlant: minMoisture=30%, optimalTemperature=28°C

CODE REUSE THROUGH INHERITANCE:

Without inheritance, each sensor would duplicate:
   string sensorID;
   float currentValue;
   float minRange;
   float maxRange;
   time_t lastReadTime;
   
   string getSensorID() const { return sensorID; }
   float getCurrentValue() const { return currentValue; }
   time_t getLastReadTime() const { return lastReadTime; }

With inheritance, this code exists ONCE in the Sensor base class and is
INHERITED by all 5 sensor types.

INHERITANCE BENEFITS IN THIS PROJECT:
✓ Eliminates code duplication across similar types
✓ Establishes clear relationships (IS-A relationships)
✓ Makes code more maintainable
✓ Enforces consistent interfaces
✓ Simplifies adding new plant/sensor/actuator types


3.4 ENCAPSULATION
================================================================================

Definition: Encapsulation bundles data (attributes) and methods together and
hides internal details through access modifiers (private, protected, public).

IMPLEMENTATION IN PROJECT:

1. DATA HIDING - SENSOR EXAMPLE:

   class Sensor {
   private:                          ← HIDDEN from outside
       string sensorID;
       float currentValue;
       float minRange;
       float maxRange;
       time_t lastReadTime;

   protected:                        ← ACCESSIBLE to derived classes
       // (inherited by MoistureSensor, etc.)

   public:                           ← ACCESSIBLE from anywhere
       float readValue();
       string getSensorType() const;
       float getCurrentValue() const;
       time_t getLastReadTime() const;
   };

BENEFIT: Outside code CANNOT:
   sensor->currentValue = -500;  // COMPILER ERROR - private
   sensor->lastReadTime = 0;     // COMPILER ERROR - private

Outside code CAN:
   float value = sensor->getCurrentValue();  // Public method
   string type = sensor->getSensorType();    // Public method

2. CONTROLLED ACCESS:

The private members are protected by getter methods:

   float getCurrentValue() const {
       return currentValue;
   }

This allows the Sensor class to:
✓ Validate data before returning it
✓ Calculate values on-the-fly
✓ Change internal representation without breaking client code
✓ Add logging or debugging without client knowledge
✓ Ensure data integrity

3. ACTUATOR ENCAPSULATION:

   class Actuator {
   private:
       string actuatorID;
       bool isActive;
       int powerLevel;

   public:
       virtual void activate(int level) = 0;
       virtual void deactivate() = 0;
       
       bool getIsActive() const { return isActive; }
       int getPowerLevel() const { return powerLevel; }
   };

Outside code can READ but not MODIFY these values directly. The ONLY way to
change the actuator state is through activate() and deactivate() methods, which
can include validation and logging.

4. PLANT ENCAPSULATION:

   class Plant {
   protected:
       float minMoisture;
       float optimalTemperature;
       bool isHealthy;
       int daysGrowing;

   public:
       bool getIsHealthy() const;
       float getCurrentMoisture() const;
       void setIsHealthy(bool health);
   };

Protected members are:
- Accessible to derived classes (TomatoPlant, LettucePlant, MaizePlant)
- NOT accessible to outside code
- Can be modified by derived classes if needed

5. BENEFITS OF ENCAPSULATION IN THIS PROJECT:

✓ Protects against invalid data:
  - Cannot set moisture to -50% (private with validation)
  - Cannot create invalid temperature readings

✓ Internal changes don't break client code:
  - If we change how currentValue is stored internally
  - Client code using getCurrentValue() still works

✓ Maintains invariants:
  - Power levels always stay 0-100%
  - Sensor values always within range
  - Plant health reflects actual conditions

✓ Facilitates logging and debugging:
  - Can add logging in getter methods
  - Can track when values change
  - Can add performance monitoring

✓ Enables future extensions:
  - Can add validation without changing interface
  - Can add caching for expensive calculations
  - Can add thread-safety without client code changes

4. DATA STRUCTURES USED
================================================================================

4.1 VECTOR (Dynamic Array)

LOCATION: Greenhouse class

   vector<Sensor*> sensors;      // 5 sensor pointers
   vector<Actuator*> actuators;  // 5 actuator pointers
   vector<Plant*> plants;        // 3 plant pointers

WHY VECTOR:
- Size is unknown at compile time
- Need random access (sensor[0], sensor[1], etc.)
- Need efficient iteration (for each sensor)
- Can dynamically add new sensors if needed

OPERATIONS:
   gh.sensors.push_back(new MoistureSensor());
   
   for (auto sensor : sensors) {
       float value = sensor->readValue();
   }

TIME COMPLEXITY:
- Access: O(1) - constant time
- Insertion: O(1) amortized
- Iteration: O(n) linear

4.2 QUEUE (First-In-First-Out)

LOCATION: Greenhouse class for alert management

   queue<Alert> alertQueue;

PURPOSE:
- Alerts are generated in order
- Should be processed in order
- Queue enforces FIFO semantics

OPERATIONS:
   // Generate alert
   Alert alert("Low moisture", "WARNING");
   alertQueue.push(alert);
   
   // Process alerts
   while (!alertQueue.empty()) {
       Alert alert = alertQueue.front();
       alertQueue.pop();
       // Process alert...
   }

BENEFITS:
✓ Guarantees first alert generated is first processed
✓ No random access (can't skip alerts)
✓ Natural data structure for event queue

4.3 STRING (Dynamic Text)

LOCATION: Throughout all classes

   string sensorID;
   string actuatorType;
   string message;

WHY STRING:
- Size varies at runtime
- Safe (manages memory automatically)
- Provides useful methods (concatenation, comparison, etc.)

4.4 USAGE SUMMARY TABLE:

   Data Structure | Location          | Purpose
   ─────────────────────────────────────────────────
   vector<Sensor*>    | Greenhouse    | Store sensors
   vector<Actuator*>  | Greenhouse    | Store actuators
   vector<Plant*>     | Greenhouse    | Store plants
   queue<Alert>       | Greenhouse    | Manage alerts
   string             | All classes   | Text storage
   vector<string>     | Logger        | Multiple CSV files

These data structures are:
✓ Standard library (STL) - no custom implementation needed
✓ Well-tested and optimized
✓ Appropriate for their use cases
✓ Support polymorphic collections (vector<Sensor*>)

   SENSOR READING
      │
      ├─→ Moisture Sensor
      │    └─ Is it < 40%?
      │        ├─ YES → Check water level
      │        │        └─ > 20%? → ACTIVATE PUMP
      │        └─ NO → DEACTIVATE PUMP
      │
      ├─→ Temperature Sensor
      │    ├─ Is it > 28°C?
      │    │  └─ YES → ACTIVATE FAN
      │    └─ Is it < 15°C?
      │       └─ YES → ACTIVATE HEATER
      │
      ├─→ Light Sensor
      │    └─ Is it < 3000 lux?

      │
      └─→ Plant Health Check
           └─ For each plant
              ├─ Moisture in range?
              ├─ Temperature in range?
              ├─ Light sufficient?
              └─ ALL OK? → Healthy : UNHEALTHY

6. FILE HANDLING & DATA PERSISTENCE

6.1 CSV FILES OVERVIEW

The project uses CSV (Comma-Separated Values) files with PIPE DELIMITERS (|)
for data persistence.

REASON FOR PIPE DELIMITER:
- Standard commas in data would break CSV parsing
- Pipe character (|) is rare in normal data
- Makes parsing more robust

6.2 CSV FILES CREATED

1. SENSOR_READINGS.CSV
   Purpose: Log all sensor measurements
   Columns: Timestamp | SensorID | SensorType | Value | Unit | MinRange | MaxRange
   Example:
   2024-05-14|12:30:45|MOIST_001|MoistureSensor|41.23|%|0|100
   2024-05-14|12:30:45|TEMP_001|TemperatureSensor|25.15|Celsius|-10|50

   Generation: Once per sensor per cycle
   Records per 10-cycle simulation: 50+ rows

2. ACTUATOR_EVENTS.CSV
   Purpose: Track all actuator activations and deactivations
   Columns: Timestamp | ActuatorID | ActuatorType | Event | PowerLevel
   Example:
   2024-05-14|12:30:46|PUMP_001|WaterPump|activate|80
   2024-05-14|12:30:47|FAN_001|CoolingFan|deactivate|0

   Generation: When actuator state changes
   Records per 10-cycle simulation: 40-60 rows

3. PLANT_STATUS.CSV
   Purpose: Track plant health and conditions over time
   Columns: Timestamp | PlantID | PlantName | PlantType | Moisture | Temperature | DaysGrowing | IsHealthy
   Example:
   2024-05-14|12:30:45|TOMATO_001|Tomato|TomatoPlant|41.23|25.15|1|Healthy
   2024-05-14|12:30:46|TOMATO_001|Tomato|TomatoPlant|39.87|26.42|1|Unhealthy

   Generation: Once per plant per cycle
   Records per 10-cycle simulation: 30 rows

4. WATER_USAGE.CSV
   Purpose: Track cumulative water consumption
   Columns: Timestamp | TotalLitersDispensed | Source
   Example:
   2024-05-14|12:30:46|0.40|WaterPump
   2024-05-14|12:30:47|0.40|WaterPump

   Generation: Once per cycle when water is used
   Records per 10-cycle simulation: 10-15 rows

5. ALERTS.CSV
   Purpose: Log system alerts and warnings
   Columns: Timestamp | Severity | Message
   Example:
   2024-05-14|12:30:46|INFO|Water pump activated - low soil moisture
   2024-05-14|12:30:47|WARNING|Plant TOMATO_001 is unhealthy - check conditions
   2024-05-14|12:31:15|CRITICAL|No sensors available for decision making

   Severity Levels: INFO, WARNING, CRITICAL
   Generation: When conditions trigger alerts
   Records per 10-cycle simulation: 15-30 rows

6. PLANT_PROFILES.CSV (Configuration)
   Purpose: Define plant parameters
   Contains: Plant ID, Name, Type, Moisture requirements, Temperature range, Light requirements
   Used by: Can be loaded at startup to configure plants

7. SYSTEM_CONFIG.CSV (Configuration)
   Purpose: Define system parameters
   Contains: Moisture threshold (40%), Temperature thresholds (15-28°C), Light threshold (3000 lux)
   Used by: Defines actuator activation thresholds

6.3 LOGGER CLASS IMPLEMENTATION

The Logger class handles all file I/O:

   class Logger {
   private:
       string sensorLogFile;
       string actuatorLogFile;
       string plantLogFile;
       string waterUsageFile;
       string alertLogFile;

   public:
       void logSensorReading(const Sensor* sensor);
       void logActuatorEvent(const Actuator* actuator, const string& event);
       void logPlantStatus(const Plant* plant);
